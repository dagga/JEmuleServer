# JEmuleServer Protocols

This document describes the network protocols implemented in JEmuleServer, based on the eDonkey2000 (ed2k) and eMule ecosystem.

## 1. eDonkey2000 (ed2k) Base Protocol

The ed2k protocol is a decentralized file-sharing protocol. JEmuleServer implements the **Server** side.

### Packet Structure
All ed2k TCP packets start with a 6-byte header:
- **Protocol (1 byte)**: `0xE3` (ED2K) or `0xC5` (eMule).
- **Size (4 bytes)**: Total size of the packet (data + 1 byte for the OpCode), Little-Endian format.
- **OpCode (1 byte)**: Message type.

### Login and Identification
Upon connection, the client sends a `LOGIN_REQUEST` (0x01) containing its name, port, and capabilities. The server responds with a series of packets to finalize the connection:
1. `SERVER_IDENT` (0x41): Contains the server hash, its IP, port, and tags (name, version, etc.).
2. `SERVER_MESSAGE` (0x38): Welcome message (MotD).
3. `LOGIN_ACCEPTED` (0x1B) or `ID_CHANGE` (0x40): Assigns the ClientID to the client.
4. `SERVER_STATUS` (0x34): Current number of users and files.

## 2. eMule Extension (0xC5)

The eMule protocol adds extended features on top of the original ed2k protocol.
- **Protocol ID**: `0xC5`.
- **Implemented OpCodes**:
    - `EMULE_INFO` (0x01) / `EMULE_INFO_ACK` (0x02): Information exchange on eMule-specific capabilities.
    - `GET_SOURCES_OBFU` (0x23): Extended source request supporting obfuscation.

## 3. Lugdunum (eServer) Extensions

The "Lugdunum" (eServer) series was the most popular ed2k server implementation in the 2000s. It introduced several proprietary extensions that became de-facto standards:

- **OpCode 0x40 (ID_CHANGE)**: Used to assign or confirm a HighID to the client. This is now the standard way to finalize a connection.
- **OpCode 0x41 (SERVER_INFO)**: Sent during the initial handshake to provide the server's identity.
- **OpCode 0x42 (SERVER_LIST)**: Used by the server to send a list of other trusted servers to the client, allowing the client to update its `server.met` file.
- **Tag 0xFB (CT_EMULE_VERSION)**: Introduced by Lugdunum to allow the server to advertise its eMule-compatible version directly in the `SERVER_IDENT` packet. This optimization prevents the client from sending an extra `EMULE_INFO` packet, speeding up the connection process.
- **Tag Extensions**: Support for new tag types (Float, Bool, Blob) and optimized 1-byte tag names for common metadata (Version, Compression, etc.).

## 4. Protocol Obfuscation (RC4)

Obfuscation aims to mask ed2k traffic to avoid filtering by Internet Service Providers (DPI).

### Handshake
The obfuscation handshake occurs before sending the first ed2k packet if the client supports it:
1. The client sends a random byte followed by 4 random bytes (used for the key), the marker `0x97`, and padding.
2. RC4 keys are generated via MD5 on the "magic byte" `0x97` and the client's random bytes.
3. The first 1024 bytes generated by the RC4 algorithm are systematically discarded to strengthen security.
4. The rest of the stream is transparently encrypted.

## 5. ZLIB Compression

To reduce bandwidth, some packets can be compressed.
- **Protocol ID**: `0xD4`.
- **Mechanism**: If the compressed payload is smaller than the original, the server uses protocol `0xD4` and compresses the data (after the OpCode).
- **Threshold**: JEmuleServer compresses packets with data size greater than 64 bytes.
